### [🟠 H-2] Rủi ro vòng lặp redirect `/users ↔ /login` khi token hết hạn nhưng cookie còn
- **Vị trí:** `apps/admin/proxy.ts:13-15` kết hợp với hành vi 401 ở tầng API (`README.md:16` mô tả "401/403 → /login").
- **Checklist:** admin-web (Access control / Session), UX.
- **Bằng chứng:**
  ```ts
  // proxy.ts
  if (hasToken && pathname === '/login') {
    return NextResponse.redirect(new URL('/users', req.url));  // có cookie → ép về /users
  }
  // còn /users gọi apiGet → backend 401 (token hết hạn) → redirect('/login') → proxy lại ép /users → loop
  ```
  `proxy` chỉ kiểm tra `req.cookies.has(TOKEN_COOKIE)` (sự *tồn tại*), không kiểm tra token còn hiệu lực.
- **Vì sao sai / rủi ro:** Cookie `admin_token` tồn tại nhưng Supabase token đã hết hạn → người dùng kẹt trong vòng lặp redirect, không vào được trang nào. (Cần xác minh thủ công khi `lib/api`/`lib/auth` được tạo lại - hiện thiếu nên chưa tái hiện được.)
- **Cách sửa đề xuất:** Khi nhận 401/403, `apiGet`/`apiFetch` phải **xoá cookie `admin_token`** trước khi `redirect('/login')` (để lần sau proxy không còn `hasToken`). Phương án mạnh hơn: trong proxy gọi `/admin/me` để verify, hoặc lưu kèm `expires_in` và check hạn ở proxy.
