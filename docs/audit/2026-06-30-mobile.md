## Phát hiện chi tiết

### [🔴 C-1] Tuya AppSecret (+ Cloud secret) nằm trong git history
- **Vị trí:** `docs/sdk/keys.txt` (đã commit + push origin trước đó).
- **Checklist:** security-secrets (secret inventory + history).
- **Bằng chứng:** file chứa `ANDROID_SECRET`/`IOS_SECRET`/`CLOUD_CLIEND_SECRET`; đã từng được commit & push lên GitHub (dù nay gitignore).
- **Vì sao sai / rủi ro:** AppSecret/Cloud secret là **server/native-only, KHÔNG được vào repo** (CLAUDE.md hard constraint). Đã lên history công khai = coi như **lộ**.
- **Cách sửa:** **rotate** AppSecret + Cloud secret trên Tuya console → `git rm --cached docs/sdk/keys.txt` + purge history (git filter-repo/BFG) → confirm `.gitignore` phủ. *(User đã chọn "để yên, tính sau" — ghi lại để không quên; vẫn là 🔴.)*

### [🟠 H-1] Android manifest thiếu permission cho pairing (Wi-Fi EZ + BLE)
- **Vị trí:** `apps/mobile/android/app/src/main/AndroidManifest.xml:3` (chỉ có `INTERNET`).
- **Checklist:** react-native (permissions) · tuya-sdk (BLE scan permissions).
- **Bằng chứng:** chỉ `<uses-permission android:name="android.permission.INTERNET" />`.
- **Vì sao sai / rủi ro:** Wi-Fi EZ cần `ACCESS_FINE_LOCATION` + `ACCESS_WIFI_STATE`/`CHANGE_WIFI_STATE`; BLE scan/pair cần `BLUETOOTH_SCAN`/`BLUETOOTH_CONNECT` (Android 12+) + legacy `BLUETOOTH`/`BLUETOOTH_ADMIN` + `ACCESS_FINE_LOCATION`. Thiếu → `startBleScan`/EZ pairing **fail runtime** (PairingScreen vừa build sẽ không hoạt động trên máy thật).
- **Cách sửa:** thêm các `<uses-permission>` trên (lib có thể merge manifest sẵn — verify khi build; nếu không, khai ở app) + **xin runtime** `BLUETOOTH_SCAN/CONNECT` + `ACCESS_FINE_LOCATION` trong luồng pairing (kèm rationale + path bị từ chối).

### [🟠 H-2] iOS Info.plist: usage string rỗng/thiếu cho Location/BLE/Local-Network
- **Vị trí:** `apps/mobile/ios/CoolBathMobile/Info.plist:37` (`NSLocationWhenInUseUsageDescription` = chuỗi RỖNG).
- **Checklist:** react-native (permissions) · tuya-sdk.
- **Bằng chứng:** `<key>NSLocationWhenInUseUsageDescription</key><string></string>`; KHÔNG có `NSBluetoothAlwaysUsageDescription`; có `NSAllowsLocalNetworking=true` nhưng KHÔNG có `NSLocalNetworkUsageDescription`.
- **Vì sao sai / rủi ro:** usage string rỗng → **App Store reject** + prompt không hiển thị lý do; thiếu BLE string → crash khi truy cập BLE; thiếu Local Network string → prompt LAN (EZ/combo) iOS 14+ thiếu mô tả. Pairing iOS không xin được quyền.
- **Cách sửa:** điền mô tả thật cho `NSLocationWhenInUseUsageDescription`; thêm `NSBluetoothAlwaysUsageDescription` + `NSLocalNetworkUsageDescription` (+ Bonjour services nếu cần cho combo/Matter sau).

### [🟠 H-3] DP-id điều khiển là placeholder, chưa map schema thật
- **Vị trí:** `apps/mobile/src/services/dp.ts:4-8` (`currentTemp:'105'`, `targetTemp:'104'`, `light:'101'`).
- **Checklist:** tuya-sdk (device control — DP phải verify theo schema thật, không đoán).
- **Bằng chứng:** comment `⚠️ PLACEHOLDER`; chỉ `targetTemp='104'` có cơ sở (README/example), current/light đặt tạm.
- **Vì sao sai / rủi ro:** publish/parse sai dp-id → **điều khiển sai hoặc lỗi `-5`** trên thiết bị thật; dashboard hiển thị sai nhiệt độ/đèn.
- **Cách sửa:** lấy **DP schema thật** của bồn (productId từ client) → cập nhật `dp.ts`; có thể đọc `getDeviceSnapshot().schemaJson`/`dpCodesJson` (lib đã trả) để map runtime. (Chặn nội dung — chung cho M2/M3.)

### [🟡 M-1] Không có Error Boundary — crash trắng màn
- **Vị trí:** `apps/mobile/App.tsx` (router switch, không bọc ErrorBoundary).
- **Checklist:** react-native (reliability).
- **Vì sao sai:** lỗi render 1 screen làm trắng toàn app, không có fallback.
- **Cách sửa:** bọc `currentScreen` bằng 1 `ErrorBoundary` (componentDidCatch) hiển thị thông báo + nút thử lại.

### [🟡 M-2] `connectDevice` bỏ qua kết quả `initSdk()`
- **Vị trí:** `apps/mobile/src/state/useAppState.ts` — `await initSdk();` (không kiểm boolean trả về).
- **Checklist:** tuya-sdk (init failure handled, not ignored).
- **Vì sao sai:** init fail (sai AppKey/region) bị nuốt; `readDevice` chạy tiếp rồi fail mơ hồ.
- **Cách sửa:** `const ok = await initSdk(); if (!ok) { báo lỗi 'SDK init failed' + dừng }`.

### [🟡 M-3] Pairing chỉ có Wi-Fi EZ, thiếu AP fallback
- **Vị trí:** `apps/mobile/src/screens/PairingScreen.tsx` (`runWifi` → `pairWifi(...,'EZ',...)`).
- **Checklist:** tuya-sdk (AP mode fallback khi EZ fail).
- **Vì sao sai:** EZ fail trên nhiều router → không có đường AP thay thế → user kẹt.
- **Cách sửa:** thêm lựa chọn/auto-fallback **AP** (`pairWifi(...,'AP',...)`) khi EZ timeout; adapter đã hỗ trợ mode.

### [🟡 M-4] Live status không throttle trước khi vào React state
- **Vị trí:** `apps/mobile/src/state/useAppState.ts` (`listenDevice` → set 3 state mỗi event).
- **Checklist:** react-native (throttle device-status updates).
- **Vì sao sai:** `onDeviceStatus` tần suất cao (nhiệt độ realtime) → nhiều re-render.
- **Cách sửa:** debounce/throttle (vd 250–500ms) hoặc gộp patch trước khi setState.

### [🟡 M-5] Promise pairing không có timeout phía JS
- **Vị trí:** `PairingScreen.runWifi`/`pickBle` → `await pairWifi/pairBleWifi` (chỉ dựa timeout native).
- **Checklist:** react-native (timeouts on pairing) · tuya-sdk (no path spinning forever).
- **Vì sao sai:** nếu native không bao giờ resolve/reject → UI kẹt 'progress'. (native truyền `timeoutSec` nên ít khả năng, nhưng nên có guard.)
- **Cách sửa:** `Promise.race` với timeout JS (vd 130s) → chuyển 'error' nếu quá hạn.

### [🔵 L-1] devId lưu AsyncStorage (plaintext)
- **Vị trí:** `apps/mobile/src/services/deviceStore.ts`.
- **Ghi chú:** `devId` KHÔNG nhạy cảm → chấp nhận. Nhưng nếu sau này lưu **token/sid**, phải dùng Keychain/Keystore (security-secrets), không AsyncStorage.

### [🔵 L-2] Nút +/- nhiệt độ publish mỗi lần bấm (không gộp)
- **Vị trí:** `apps/mobile/src/screens/HomeScreen.tsx:272,279` → `state.setTargetTemp(...)` mỗi tap → `publishDps`.
- **Ghi chú:** bấm nhanh = spam `publishDps`. ±1/tap nên rủi ro thấp; cân nhắc debounce/coalesce khi gửi.
